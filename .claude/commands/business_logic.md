# Business Logic Audit

Audit business logic consistency across backend (source of truth) and all clients (web-client).
Do not guess ‚Äî read the actual files before reporting any finding.

## Scope

**Backend (source of truth)**
- `task-service/src/main/java/com/demo/task/service/` ‚Äî all service classes
- `task-service/src/main/java/com/demo/task/model/` ‚Äî all entity classes
- `common/src/main/java/com/demo/common/dto/` ‚Äî all enums and DTOs

**Frontend (must mirror backend rules)**
- `web-client/src/api/` ‚Äî API layer and type definitions
- `web-client/src/pages/` ‚Äî page-level forms and validation
- `web-client/src/components/` ‚Äî UI components with guards

## Business Rules Reference (backend-defined)

### Enums ‚Äî allowed values

| Enum | Values |
|---|---|
| `TaskStatus` | `TODO`, `IN_PROGRESS`, `DONE` |
| `TaskType` | `FEATURE`, `BUG_FIXING`, `TESTING`, `PLANNING`, `TECHNICAL_DEBT`, `DOCUMENTATION`, `OTHER` |
| `TaskPhaseName` | `BACKLOG`, `TODO`, `IN_PROGRESS`, `IN_REVIEW`, `TESTING`, `DONE`, `RELEASED` |
| `TaskParticipantRole` | `CREATOR`, `ASSIGNEE`, `VIEWER`, `REVIEWER` |
| `WorkType` | `DEVELOPMENT`, `TESTING`, `CODE_REVIEW`, `DESIGN`, `PLANNING`, `DOCUMENTATION`, `DEPLOYMENT`, `MEETING`, `OTHER` |
| `TimelineState` | `PLANNED_START`, `PLANNED_END`, `REAL_START`, `REAL_END` |

### Task rules

- `title` ‚Äî required
- `projectId` ‚Äî required; must reference an existing project
- `phaseId` ‚Äî optional on create; falls back to project's `defaultPhaseId`; must not be null on update
- `assignedUserId` ‚Äî required; must reference an existing user
- `plannedStart` ‚Äî required; must be before `plannedEnd`
- `plannedEnd` ‚Äî required; must be after `plannedStart`
- `progress` ‚Äî integer 0‚Äì100
- `taskCode` ‚Äî auto-generated by backend; clients must never allow manual input
- Deletion blocked if task has any comments ‚Üí `RelatedEntityActiveException` (HTTP 409)

### TaskProject rules

- `name` ‚Äî required
- `taskCodePrefix` ‚Äî defaults to `"TASK_"` if blank/null
- Deletion blocked if project has any tasks ‚Üí HTTP 409

### TaskPhase rules

- `name` ‚Äî must be one of `TaskPhaseName` enum values
- `projectId` ‚Äî required; must reference an existing project
- Deletion blocked if any tasks are assigned to this phase ‚Üí HTTP 409

### TaskParticipant rules

- `CREATOR` role is set only at task creation; clients must never offer it as an option when adding participants manually
- Each user can hold each role at most once per task (unique user+role+task constraint)
- `ASSIGNEE` is singular per task; setting a new assignee automatically replaces the old one

### TaskTimeline rules

- `PLANNED_START` must be before `PLANNED_END` when both are set
- `REAL_START` must be before `REAL_END` when both are set
- Clients should enforce the ordering before submitting

### TaskPlannedWork rules

- Can only be created when task status is `TODO`; clients must disable or hide the creation form otherwise
- One entry per `workType` per task; clients must not offer already-used work types in the dropdown
- Immutable after creation ‚Äî no edit or delete UI should be shown

### TaskBookedWork rules

- Multiple entries per `workType` per task are allowed
- `bookedHours` defaults to 0 if omitted; clients should treat blank as 0

### ProjectNotificationTemplate rules

- `{placeholder}` tokens must match the `TemplatePlaceholder` enum; clients should display the recognized list to users

### Soft-delete pattern

- All entities use soft-delete (`deleted_at`); no physical deletes exist
- Deleted entities must not be shown in any list or selectable in any dropdown

## Audit Instructions

For each rule category below, read the relevant backend service and the corresponding frontend files, then report:

- **Status**: PASS / FAIL / WARN / N/A
- **Finding**: what was confirmed or found wrong
- **File**: exact path and line number(s)
- **Recommendation**: specific fix with before/after if FAIL or WARN

### 1 ‚Äî Enum completeness (CRITICAL)
- Do frontend type definitions and dropdowns include exactly the same values as the backend enums?
- Are any backend enum values missing from the frontend, or are there extra frontend-only values?

### 2 ‚Äî Required-field validation (CRITICAL)
- Does the frontend validate all required fields before submitting (title, projectId, assignedUserId, plannedStart, plannedEnd)?
- Are date ordering rules (plannedStart < plannedEnd, realStart < realEnd) checked on the frontend?

### 3 ‚Äî Read-only field protection (HIGH)
- Is `taskCode` treated as display-only everywhere in the frontend (never in a form input)?
- Is the `CREATOR` participant role absent from the "add participant" role dropdown?

### 4 ‚Äî Status-gated UI (HIGH)
- Is the `TaskPlannedWork` creation form disabled or hidden when task status is not `TODO`?
- Are already-used `workType` values removed from the planned work creation dropdown?

### 5 ‚Äî Deletion guards (HIGH)
- Does the frontend surface a meaningful error (not a generic toast) when the backend returns HTTP 409 `RelatedEntityActiveException`?
- Are delete buttons guarded by a confirmation dialog?

### 6 ‚Äî Assignee uniqueness (MEDIUM)
- Does the frontend treat assignee as a single-select field (not multi-select)?
- Is there any UI that suggests multiple assignees can exist simultaneously?

### 7 ‚Äî Soft-delete visibility (MEDIUM)
- Does the frontend filter out or never receive deleted entities in lists and dropdowns?
- Are there any client-side caches or local state that could show stale deleted items?

### 8 ‚Äî Timeline ordering (MEDIUM)
- Does the timeline date-picker enforce PLANNED_START < PLANNED_END and REAL_START < REAL_END before submit?

### 9 ‚Äî Notification template placeholders (LOW)
- Does the template editor list or suggest only recognized `TemplatePlaceholder` values?

---

## Output Format

End the report with:

1. **FAIL list** (must fix ‚Äî backend and frontend are inconsistent)
2. **WARN list** (frontend allows something the backend will reject, or misses a guard)
3. **PASS list** (rules correctly mirrored on both sides)
4. **Overall verdict**: CONSISTENT / PARTIALLY CONSISTENT / INCONSISTENT
