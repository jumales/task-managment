# Frontend overview

**Module:** `web-client/`
**Framework:** React 18.3 + TypeScript 5.4
**Build tool:** Vite 5.2
**UI library:** Ant Design 5.17
**Dev server port:** 3000

The frontend is a single-page application served by Nginx inside Docker. In development it runs via `vite dev` on port 3000. All API calls go through the API Gateway at `http://localhost:8080`.

## Authentication

The app uses **Keycloak JS** with the **PKCE** (Proof Key for Code Exchange) method. On startup:

1. `main.tsx` initializes Keycloak with `onLoad: 'login-required'`
2. If no valid session exists, the user is redirected to the Keycloak login page
3. After successful login, Keycloak issues a JWT stored in the browser session
4. An **Axios interceptor** attaches the token as `Authorization: Bearer <token>` on every API request
5. Tokens are silently refreshed before expiry

## API client layer

All API calls are isolated in `src/api/`. Each file exports typed async functions wrapping Axios. Components and hooks never call Axios directly.

```typescript
// taskApi.ts
export const getTasks = (params: TaskFilterParams) =>
  apiClient.get<PageResponse<TaskResponse>>('/api/v1/tasks', { params });

export const createTask = (request: TaskRequest) =>
  apiClient.post<TaskResponse>('/api/v1/tasks', request);
```

## Custom hooks

Data fetching and mutation logic lives in `src/hooks/`. Pages and components call hooks — they do not contain fetch logic themselves.

| Hook | Purpose |
|---|---|
| `useTaskDetailData` | Fetch full task details |
| `useTaskRealtime` | Subscribe to WebSocket task updates |
| `useTaskComments` | List and post comments |
| `useTaskAttachments` | Upload and remove attachments |
| `useTaskParticipants` | Manage task team members |
| `useTaskPlannedWork` | Record estimated hours |
| `useTaskBookedWork` | Log and update actual hours |
| `useTaskPhaseChange` | Transition task to a different phase |
| `useTaskTimeline` | Manage task timelines |
| `useAvatarBlobUrl` | Lazy-load user avatar images |

## Internationalisation

All user-visible strings use `i18next`. Locale files live in `src/i18n/locales/`. To add a new language, add a new locale file and register it in the i18n initializer.
