# Database Structure

Each microservice owns its own isolated PostgreSQL database. There are no cross-service foreign keys. Schema changes are managed exclusively via Flyway migrations (`V{n}__{description}.sql`).

---

## Design Conventions

| Convention | Detail |
|---|---|
| **Soft deletes** | All mutable entities have `deleted_at TIMESTAMPTZ`; records are never hard-deleted |
| **Partial indexes** | Indexes on soft-delete tables use `WHERE deleted_at IS NULL` to stay small and avoid blocking unique constraints for deleted rows |
| **Unique partial indexes** | Used instead of `UNIQUE` constraints on soft-delete tables (e.g. `uq_tasks_task_code`) |
| **UUID v7 primary keys** | Time-ordered UUIDs generated by Hibernate `@UuidGenerator(style = TIME)`; embed creation timestamp (used by archive scheduler) |
| **No cross-service FKs** | Each service has its own database; references to other services' entities are plain UUID columns |
| **Archive schemas** | `task-service` and `audit-service` each have an `archive` schema with dynamically-created monthly tables (`{table}_{YYYYMM}`) |
| **Idempotency tables** | `processed_kafka_events` in audit-service and notification-service deduplicates Kafka consumption using `(event_id, consumer_group)` unique index |
| **Outbox pattern** | `outbox_events` in task-service guarantees at-least-once Kafka delivery; partial index on `published = FALSE` keeps polling fast |
