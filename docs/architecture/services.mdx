## Common module

All services depend on the `common` Maven module. It provides:

- **DTOs** — `TaskRequest`, `TaskResponse`, `UserDto`, `PageResponse<T>`, and all other shared request/response types
- **Enums** — `TaskStatus`, `TaskType`, `TaskPhaseName`, `TaskChangeType`, `WorkType`, `UserEventType`, and more
- **Events** — `TaskChangedEvent`, `UserEvent` (Kafka message payloads)
- **Security** — `SecurityConfig`, `JwtAuthConverter`, `FeignAuthInterceptor`, `MdcFilter`
- **Web utilities** — `ControllerLoggingAspect`, `LoggingAuthenticationEntryPoint`, `ResponseCode`

<Note>
Any DTO or event used by two or more services **must** live in `common`. Never duplicate these classes across services.
</Note>

## Config Server

The config-server module is a Spring Cloud Config Server running at port 8888. It uses a **native filesystem backend** — the `config-repo/` directory in the monorepo root is the single source of truth for all shared and service-specific configuration.

### Config resolution

Every application service (except eureka-server) fetches its merged configuration on startup:

```
config-repo/application.yml          ← shared by ALL services
config-repo/{service-name}.yml       ← service-specific overrides
```

The merged result is what a service actually sees at runtime. Services use `spring.config.import` (not the older `bootstrap.yml`) to pull from Config Server:

```yaml
# Each service's local application.yml contains only:
spring:
  application:
    name: task-service   # Config Server uses this to find task-service.yml
  config:
    import: "optional:configserver:http://${CONFIG_SERVER_URL:localhost:8888}"
```

The `optional:` prefix allows services to start in isolation (e.g. unit test runs) even when Config Server is not running.

### Startup order

Config Server must be healthy before any application service starts. It runs as a Docker container and is health-checked before Eureka:

```
Postgres / Kafka / Keycloak / MinIO / Redis (Docker)
  → config-server (Docker, health-checked at /actuator/health)
  → eureka-server (local process)
  → all application services (local processes)
```

### Verify config is loaded

```bash
# Check merged config for task-service
curl http://localhost:8888/task-service/default

# Check which property sources a running service resolved
curl http://localhost:<port>/actuator/env | jq '.propertySources[].name'
# Should include: "configserver:..."
```

### Adding or changing shared config

Edit `config-repo/application.yml` for a change that affects all services. Edit `config-repo/{service}.yml` for a single-service change. Restart the affected service — no code changes required.

## Eureka HA setup

Two Eureka peers replicate registrations to each other:

```yaml
# peer1 registers with peer2
eureka.client.serviceUrl.defaultZone: http://eureka-peer2:8762/eureka/

# peer2 registers with peer1
eureka.client.serviceUrl.defaultZone: http://eureka-peer1:8761/eureka/
```

Application services register with both peers for fault tolerance. If one Eureka instance restarts, clients continue discovering services from the other.
