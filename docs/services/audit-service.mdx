# Audit service

**Module:** `audit-service/`
**Database:** `audit_db` (PostgreSQL)
**Kafka:** Consumes from `task-changed`

The audit service maintains an **immutable, append-only history** of all task changes. It never modifies or deletes audit records. Its database is a pure read model built from `TaskChangedEvent` messages.

## Audit tables

Each change type is recorded in its own table for clean separation and efficient querying.

| Table | Populated by change type | Columns |
|---|---|---|
| `status_audit_records` | `STATUS_CHANGED` | taskId, fromStatus, toStatus, changedBy, changedAt |
| `phase_audit_records` | `PHASE_CHANGED` | taskId, fromPhase, toPhase, changedBy, changedAt |
| `comment_audit_records` | `COMMENT_ADDED` | taskId, commentId, content, authorId, createdAt |
| `planned_work_audit_records` | `PLANNED_WORK_CREATED` | taskId, userId, workType, hours, recordedAt |
| `booked_work_audit_records` | `BOOKED_WORK_CREATED`, `BOOKED_WORK_UPDATED`, `BOOKED_WORK_DELETED` | taskId, userId, workType, hours, action, recordedAt |

## Kafka consumer

Events are consumed with manual acknowledgement from `task-changed`. The service dispatches on `changeType` and writes to the appropriate audit table in a single insert. There is no complex business logic — every event maps directly to an insert.

Every consumer guards its work with a dedup check against `processed_kafka_events` (`(event_id, consumer_group)` unique index). Duplicate deliveries — which happen whenever the task-service outbox publisher crashes after a Kafka ACK but before its `published = true` commit — are silently skipped so no duplicate audit row is written. The same guard protects the `TaskLifecycleConsumer` archive path on the `task-events` topic.

## Controllers

| Endpoint | Method | Description |
|---|---|---|
| `/api/v1/audit/tasks/{id}/statuses` | `GET` | Paginated status transition history for a task |
| `/api/v1/audit/tasks/{id}/phases` | `GET` | Paginated phase transition history |
| `/api/v1/audit/tasks/{id}/comments` | `GET` | Paginated comment history |
| `/api/v1/audit/tasks/{id}/planned-work` | `GET` | Planned work log history |
| `/api/v1/audit/tasks/{id}/booked-work` | `GET` | Actual work log history |
| `/api/v1/dlq/status` | `GET` | DLT consumer lag per dead-letter topic (admin only) |

All endpoints return paginated results ordered chronologically (oldest first).

## DLQ monitoring

Failed Kafka messages are forwarded to `<topic>.DLT` after the bounded retry budget is exhausted
(`KafkaDlqConfig`). Audit-service exposes three complementary monitoring surfaces for these topics:

| Surface | Purpose |
|---|---|
| `GET /api/v1/dlq/status` | Returns `dltConsumerLag` — unconsumed messages per DLT. Drops to 0 once messages are processed or replayed. |
| `/actuator/health/dlt` | Reports `DOWN` whenever any DLT has consumer lag &gt; 0. Suitable as a Kubernetes readiness probe gate. |
| `kafka.dlt.consumer.lag` gauge | Micrometer gauge tagged by topic, scraped by Prometheus. Refreshed every 30 s. |

Lag is computed as `end_offset − committed_offset` for the DLT's dedicated consumer group
(constants in `KafkaTopics`). Unlike raw end offsets, this signal drops back to zero after
drain or replay, so dashboards and alerts can rely on it.

Recommended Prometheus alert rule:

```yaml
- alert: DltMessagesPending
  expr: kafka_dlt_consumer_lag > 0
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Dead-letter topic {{ $labels.topic }} has {{ $value }} unprocessed messages"
```

## Audit archiving

When task-service archives a task it publishes a `TASK_ARCHIVED` event on the `task-events` topic. The audit service consumes this event and moves all audit records for that task into monthly archive tables in the `archive` PostgreSQL schema.

| Main table | Archive table |
|---|---|
| `audit_records` | `archive.audit_records_YYYYMM` |
| `comment_audit_records` | `archive.comment_audit_records_YYYYMM` |
| `phase_audit_records` | `archive.phase_audit_records_YYYYMM` |
| `booked_work_audit_records` | `archive.booked_work_audit_records_YYYYMM` |
| `planned_work_audit_records` | `archive.planned_work_audit_records_YYYYMM` |

The YYYYMM suffix comes from the `archiveMonth` field of the `TASK_ARCHIVED` event (derived from the task's creation date, not close date).

A nightly scheduler drops audit archive tables older than `ttl.audit.archive-retention-months` (default 24 months).

## Design notes

- Audit records are **never updated** — they are a compliance-grade event log.
- Records are archived (not deleted) — the archive schema preserves full history for the configured retention window.
- The service does not call any other service. All data it needs is carried in the event payload.
- Queries are simple `findByTaskId` lookups — no complex joins or aggregations.
