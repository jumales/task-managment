# Notification service

**Module:** `notification-service/`
**Database:** `notification_db` (PostgreSQL)
**Kafka:** Consumes from `task-changed`

The notification service reacts to task changes and delivers alerts through two channels:
1. **Email** via SMTP (MailHog in dev)
2. **WebSocket push** to connected browser clients via SockJS/STOMP

## Kafka consumer

The service listens to the `task-changed` topic with manual acknowledgement. Every `TaskChangedEvent` triggers:

1. Resolve the task's assignee user (via `user-service` Feign call, cached in Redis)
2. Render the notification subject and body using the project's `ProjectNotificationTemplate`
3. Send the email via SMTP
4. Push a WebSocket message to `/topic/tasks/{taskId}`
5. Store a `Notification` record in `notification_db` for audit

## Notification templates

Each project can have a custom `ProjectNotificationTemplate` configured via task-service. Templates support placeholders:

| Placeholder | Replaced with |
|---|---|
| `{taskUrl}` | Direct URL to the task in the frontend |
| `{assigneeName}` | Full name of the task assignee |
| `{taskTitle}` | Task title |
| `{changeType}` | Human-readable change description |

If no template is configured for a project, a default system template is used.

## WebSocket endpoint

Connected frontend clients subscribe to `/topic/tasks/{taskId}` to receive real-time updates for a specific task. The subscription is established via SockJS at `/ws/tasks/**` (proxied by the API Gateway).

```
Browser subscribes to /topic/tasks/abc-123
  ↓
notification-service publishes message on TaskChangedEvent
  ↓
Browser receives update, React component re-fetches task data
```

## Change types that trigger notifications

All `TaskChangeType` values produce a notification. The email content varies by type (e.g. "Status changed from TODO to IN_PROGRESS", "New comment added").

## FCM push notifications to mobile clients

When `fcm.enabled=true` (configured via `FCM_ENABLED` env var), the service fans out a data-only FCM push to all registered devices of affected users on every `TaskChangedEvent`.

### Fan-out targets

For each event, affected users = `{assignedUserId}` + `{workLogUserId}` (if set) + all task participants (fetched from task-service via Feign). If task-service is unavailable, the push degrades to the direct assignee/work-log user only.

### Push payload

```json
{
  "changeType": "STATUS_CHANGED",
  "taskId": "uuid",
  "projectId": "uuid"
}
```

The Android client uses `changeType` to route the push to the appropriate screen.

### Token lifecycle

Tokens are stored in `device_tokens` (soft-delete on removal). When FCM returns `UNREGISTERED` or `INVALID_ARGUMENT` for a token, the service automatically soft-deletes it to avoid future wasted sends.

### Configuration

| Property | Default | Description |
|---|---|---|
| `fcm.enabled` | `false` | Enable/disable FCM delivery |
| `fcm.credentials-json` | _(empty)_ | Firebase service-account JSON (set via env) |
| `fcm.dry-run` | `false` | Log sends without actually calling FCM |

## Controllers

| Endpoint | Method | Description |
|---|---|---|
| `/api/v1/notifications/tasks/{taskId}` | `GET` | Notification history for a task |
| `/api/v1/device-tokens` | `POST` | Register or refresh a device token |
| `/api/v1/device-tokens/{oldToken}` | `PUT` | Rotate a device token |
| `/api/v1/device-tokens/{token}` | `DELETE` | Remove a device token (on logout) |
| `/api/v1/device-tokens/me` | `GET` | List active tokens for the caller |

## Notification cleanup

`NotificationCleanupScheduler` runs nightly at 05:00 and hard-deletes `Notification` records older than `ttl.notification.retention-days` (default 365 days). This prevents the `notifications` table from growing unbounded over time.

## Service-to-service auth

The notification service uses the **client credentials** flow (not request token forwarding) to call `user-service`. It obtains a Keycloak service account token and includes it as Bearer on outbound Feign requests.
